# 기능 명세: User 및 게시글 CRUD with JWT 인증

**기능 브랜치**: `001-user-post-crud`
**작성일**: 2026-03-30
**상태**: 초안

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 회원가입 및 로그인 (우선순위: P1)

신규 사용자가 이메일과 비밀번호로 회원가입하고, 로그인 시 JWT 토큰을 발급받아
이후 API 요청에 사용한다.

**이 우선순위인 이유**: 인증 없이는 보호된 기능을 전혀 사용할 수 없으므로 모든 기능의 전제 조건이다.

**독립 테스트**: 회원가입 → 로그인 → JWT 토큰 수령만으로 독립적으로 검증 가능하며,
토큰 발급 자체가 MVP 가치를 제공한다.

**인수 시나리오**:

1. **Given** 미가입 이메일, **When** 이메일·비밀번호로 회원가입 요청, **Then** 201 응답과 함께 사용자 정보 반환
2. **Given** 가입된 이메일·비밀번호, **When** 로그인 요청, **Then** 200 응답과 함께 JWT 액세스 토큰 반환
3. **Given** 잘못된 비밀번호, **When** 로그인 요청, **Then** 401 응답 반환
4. **Given** 이미 사용 중인 이메일, **When** 회원가입 요청, **Then** 409 응답 반환

---

### 사용자 스토리 2 - 게시글 목록 조회 및 상세 조회 (우선순위: P2)

비인증 사용자를 포함한 누구나 게시글 목록과 특정 게시글 상세 내용을 조회할 수 있다.

**이 우선순위인 이유**: 읽기 전용 기능이므로 인증과 무관하게 독립적으로 제공 가능하다.

**독립 테스트**: 토큰 없이 목록 조회 및 상세 조회 API 호출만으로 독립적으로 검증 가능하다.

**인수 시나리오**:

1. **Given** 게시글이 존재, **When** 비인증 상태로 목록 조회, **Then** 200 응답과 함께 게시글 목록 반환
2. **Given** 존재하는 게시글 ID, **When** 상세 조회, **Then** 200 응답과 함께 게시글 내용 반환
3. **Given** 존재하지 않는 게시글 ID, **When** 상세 조회, **Then** 404 응답 반환

---

### 사용자 스토리 3 - 토큰 갱신 (우선순위: P3)

액세스 토큰이 만료된 사용자가 리프레시 토큰으로 새 액세스 토큰을 발급받아
재로그인 없이 서비스를 계속 이용할 수 있다.

**이 우선순위인 이유**: 인증 UX를 크게 개선하지만 핵심 CRUD 기능과는 독립적으로 구현 가능하다.

**독립 테스트**: 로그인 → 리프레시 토큰으로 액세스 토큰 재발급 흐름으로 독립 검증 가능하다.

**인수 시나리오**:

1. **Given** 유효한 리프레시 토큰, **When** 토큰 갱신 요청, **Then** 200 응답과 함께 새 액세스 토큰 반환
2. **Given** 만료된 리프레시 토큰, **When** 토큰 갱신 요청, **Then** 401 응답 반환 (재로그인 필요)
3. **Given** 로그아웃된 사용자의 리프레시 토큰, **When** 갱신 요청, **Then** 401 응답 반환

---

### 사용자 스토리 4 - 사용자 프로필 수정 및 회원 탈퇴 (우선순위: P4)

인증된 사용자가 자신의 닉네임 등 프로필 정보를 수정하거나 계정을 탈퇴할 수 있다.

**이 우선순위인 이유**: 핵심 기능(인증, CRUD)이 완성된 후 제공하는 부가 기능이다.

**독립 테스트**: 로그인 → 프로필 수정 / 회원 탈퇴 흐름으로 독립 검증 가능하다.

**인수 시나리오**:

1. **Given** 유효한 JWT, **When** 닉네임 변경 요청, **Then** 200 응답과 함께 수정된 프로필 반환
2. **Given** 유효한 JWT, **When** 회원 탈퇴 요청, **Then** 204 응답 반환 및 이후 해당 계정 로그인 불가
3. **Given** 토큰 없음, **When** 프로필 수정 또는 탈퇴 요청, **Then** 401 응답 반환

---

### 사용자 스토리 5 - 게시글 작성·수정·삭제 (우선순위: P5)

인증된 사용자가 게시글을 작성하고, 자신이 작성한 게시글을 수정하거나 삭제할 수 있다.

**이 우선순위인 이유**: P1(인증)과 P2(조회)가 완성된 후에야 의미 있는 쓰기 기능이 된다.

**독립 테스트**: 로그인 → 게시글 작성 → 수정 → 삭제 흐름으로 독립적으로 검증 가능하다.

**인수 시나리오**:

1. **Given** 유효한 JWT 토큰, **When** 제목·내용으로 게시글 작성, **Then** 201 응답과 함께 게시글 반환
2. **Given** 본인 게시글과 유효한 JWT, **When** 수정 요청, **Then** 200 응답과 함께 수정된 게시글 반환
3. **Given** 타인 게시글과 유효한 JWT, **When** 수정 또는 삭제 요청, **Then** 403 응답 반환
4. **Given** 유효한 JWT 토큰, **When** 본인 게시글 삭제, **Then** 204 응답 반환
5. **Given** 토큰 없음, **When** 게시글 작성 요청, **Then** 401 응답 반환

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 이메일·비밀번호로 회원가입을 허용해야 한다
- **FR-002**: 시스템은 이메일 중복 여부를 검증해야 한다
- **FR-003**: 시스템은 로그인 성공 시 JWT 액세스 토큰과 리프레시 토큰을 함께 발급해야 한다
- **FR-004**: 시스템은 만료되거나 유효하지 않은 토큰의 요청을 거부해야 한다
- **FR-005**: 사용자는 유효한 리프레시 토큰으로 새 액세스 토큰을 발급받을 수 있어야 한다
- **FR-006**: 사용자는 자신의 닉네임 등 프로필 정보를 수정할 수 있어야 한다
- **FR-007**: 사용자는 계정을 탈퇴할 수 있어야 하며, 탈퇴 후 해당 계정으로 로그인이 불가해야 한다
- **FR-008**: 비인증 사용자는 게시글 목록 및 상세 조회가 가능해야 한다
- **FR-009**: 인증된 사용자만 게시글을 작성할 수 있어야 한다
- **FR-010**: 사용자는 자신이 작성한 게시글만 수정·삭제할 수 있어야 한다
- **FR-011**: 게시글 목록은 최신순으로 정렬되어야 한다
- **FR-012**: 시스템은 비밀번호를 안전하게 해싱하여 저장해야 한다

### 핵심 엔티티

- **User**: 서비스 사용자. 이메일(고유), 비밀번호(해싱), 닉네임을 가진다
- **Post**: 게시글. 제목, 내용, 작성자(User), 작성일시, 수정일시를 가진다

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 사용자는 2분 이내에 회원가입과 첫 로그인을 완료할 수 있다
- **SC-002**: 비인증 사용자도 게시글 목록을 지연 없이 조회할 수 있다
- **SC-003**: 인증 오류(401, 403)는 명확한 메시지와 함께 반환된다
- **SC-004**: 게시글 작성·수정·삭제는 본인 확인 후에만 성공한다

## 가정사항

- 이메일 인증(메일 발송)은 v1 범위 밖이다
- 게시글 페이지네이션은 기본 구현으로 포함한다
